# GeoHash Specification

AI向けドメインナレッジ。GeoHash の仕様をまとめる。

## 概要

GeoHash は緯度/経度を短い文字列にエンコードする空間インデックスシステム。
矩形セルで地球表面を分割し、文字列のプレフィックス一致で空間的な近傍性を表現できる（プレフィックスが長く一致するほど近い）。

- 参考: https://en.wikipedia.org/wiki/Geohash
- 座標系: WGS 84 (EPSG:4326)
- セル形状: 矩形（緯度方向と経度方向で異なるサイズ）

## Precision（精度）とセルサイズ

| Precision | 緯度幅   | 経度幅   | 概算サイズ         |
| --------- | -------- | -------- | ------------------ |
| 1         | 45°      | 45°      | ~5,000 km          |
| 2         | 5.625°   | 11.25°   | ~1,250 km × 625 km |
| 3         | 1.40625° | 1.40625° | ~156 km            |
| 4         | 0.17578° | 0.35156° | ~39 km × 19.5 km   |
| 5         | 0.04395° | 0.04395° | ~4.9 km            |
| 6         | 0.00549° | 0.01099° | ~1.2 km × 0.6 km   |
| 7         | 0.00137° | 0.00137° | ~153 m             |
| 8         | 0.00017° | 0.00034° | ~38 m × 19 m       |
| 9         | 0.00004° | 0.00004° | ~4.8 m             |

### セルサイズの計算

1文字あたり5ビット。偶数ビットが経度、奇数ビットが緯度を二分割する。

```
totalBits = 5 * precision
lonBits = ceil(totalBits / 2)
latBits = floor(totalBits / 2)
latSize = 180 / 2^latBits
lonSize = 360 / 2^lonBits
```

奇数precisionでは緯度と経度のセルサイズが等しくなり、偶数precisionでは経度が緯度の2倍になる。

## Base-32 文字セット

```
0123456789bcdefghjkmnpqrstuvwxyz
```

- 32文字（5ビット）
- **除外文字**: `a`, `i`, `l`, `o`（視認性のため）
- 大文字は使用しない

## エンコード

経度と緯度を交互に二分割し、ビット列を5ビットずつbase-32文字に変換する。

1. 初期範囲: 緯度 `[-90, 90]`, 経度 `[-180, 180]`
2. 経度から開始し、交互に処理
3. 各ステップで中間値 `mid` を計算:
   - 座標 ≥ mid → ビット1（上半分の範囲を新範囲に）
   - 座標 < mid → ビット0（下半分の範囲を新範囲に）
4. 5ビット蓄積するたびにbase-32文字を出力
5. 指定precision分の文字が生成されるまで繰り返し

### エンコード例（東京駅: 35.6812, 139.7671）

```
precision 1: "x"
precision 3: "xn7"
precision 5: "xn76u"
precision 7: "xn76urg"
precision 9: "xn76urgtq"
```

## デコード

エンコードの逆操作。各文字を5ビットに変換し、交互に経度/緯度の範囲を狭めていく。
結果としてバウンディングボックス（latMin/Max, lonMin/Max）が得られ、その中心がセルの代表座標となる。

## プレフィックスの階層性

GeoHash の重要な特性として、短いコードは長いコードのプレフィックスになる。

```
"x"       → precision 1（大きな矩形）
"xn"      → precision 2（上記の部分領域）
"xn7"     → precision 3（さらに細かい部分領域）
"xn76u"   → precision 5
```

あるコードのプレフィックスが一致する = 同じ上位セルに含まれる。

**注意**: プレフィックス一致は近傍性の十分条件ではない。セル境界をまたぐ場合、非常に近い2点でもプレフィックスが大きく異なることがある（edge case）。

## セル形状

各セルは緯度/経度の範囲で定義される矩形。4頂点で構成される。

```
NW ──── NE
│        │
│        │
SW ──── SE
```

## GeoHex との主な違い

| 項目               | GeoHex v3              | GeoHash                            |
| ------------------ | ---------------------- | ---------------------------------- |
| セル形状           | 正六角形               | 矩形                               |
| 座標系             | メルカトル投影ベース   | 緯度/経度直接分割                  |
| 文字セット         | 英大小文字 + 数字(0-8) | 数字 + 小文字(一部除外)            |
| レベル範囲         | 0-15                   | 1-12                               |
| レベルとコード長   | `level + 2`            | `precision`（= コード長そのもの）  |
| プレフィックス階層 | なし                   | あり（短いコードは長いコードの親） |
| 中心からの距離     | 均等（六角形の特性）   | 不均等（矩形の対角と辺で異なる）   |
| 隣接セル数         | 6                      | 8（辺4 + 角4）                     |

## 特記事項

- **日付変更線**: 経度が ±180° を跨ぐ場合、ラッピング処理が必要。
- **極付近**: 緯度 ±90° ちょうどだと一部の計算で問題が生じるため、±89.999999° 程度でクランプするのが一般的。
